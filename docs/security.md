# SecureZip 安全设计文档

## 加密架构概述

SecureZip 采用多层安全架构，核心加密逻辑全部在 Rust 中实现 (非 JVM / 非 Kotlin)，
避免 Java 反编译风险。

```
用户密码
    |
    v
Argon2id(m=64MB, t=3, p=4) + 随机 Salt(128-bit)
    |
    v
256-bit 主密钥 (master_key)
    |
    +--- HKDF("zbak-verify")  ---> 验证子密钥 ---> 密码验证块 (即时错误检测)
    |
    +--- HKDF("zbak-file-0")  ---> 文件 0 子密钥 ---> AES-256-GCM(随机nonce) + Zstd数据
    +--- HKDF("zbak-file-1")  ---> 文件 1 子密钥 ---> AES-256-GCM(随机nonce) + Zstd数据
    +--- HKDF("zbak-file-N")  ---> 文件 N 子密钥 ---> AES-256-GCM(随机nonce) + Zstd数据
    |
    +--- HKDF("zbak-index")   ---> 索引子密钥  ---> AES-256-GCM(索引区文件名/路径)
```


## zbak 格式安全设计

### 1. 密钥派生: Argon2id

**算法**: Argon2id (OWASP 2024 推荐的密码哈希算法)

**参数**:

| 参数 | 值 | 说明 |
|------|-----|------|
| 内存 (m) | 65536 (64 MB) | 高内存开销，抗 GPU/ASIC 暴力破解 |
| 迭代 (t) | 3 | 三次迭代增加计算时间 |
| 并行度 (p) | 4 | 四线程并行 |
| 输出长度 | 32 字节 (256 bit) | AES-256 所需密钥长度 |
| Salt | 128 bit 随机 | 存储在文件头 [8-23] 字节 |
| 版本 | v0x13 (1.3) | Argon2 最新规范版本 |

**安全等级**: 在消费级硬件上，单次密钥派生耗时约 200-500ms。
攻击者即使使用 GPU 集群，每秒也只能尝试极少量密码组合。

### 2. HKDF 子密钥派生

**算法**: HKDF-SHA256 (NIST SP 800-56C)

从一个主密钥派生多个独立子密钥：

| 子密钥 | HKDF info | 用途 |
|--------|-----------|------|
| 验证子密钥 | `"zbak-verify"` | 密码验证块 |
| 文件 i 子密钥 | `"zbak-file-{i}"` | 第 i 个文件的加密 |
| 索引子密钥 | `"zbak-index"` | 索引区加密 (文件名隐私) |

HKDF salt 使用固定值 `"zbak-hkdf-v1"`，满足 NIST 规范要求。

**安全意义**: 每个文件使用独立子密钥。即使攻击者通过侧信道获取了某个文件的子密钥，
也无法推导出主密钥或其他文件的子密钥。

### 3. AES-256-GCM 加密

**算法**: AES-256-GCM (AEAD — 认证加密附加数据)

| 参数 | 值 | 说明 |
|------|-----|------|
| 密钥长度 | 256 bit | 来自 HKDF 派生的子密钥 |
| Nonce | 96 bit (12 字节) | 每次加密随机生成 |
| GCM Tag | 128 bit (16 字节) | 用于完整性验证 |

GCM 模式同时提供**保密性**和**完整性**保护：
- 加密保证只有持有密钥者能读取数据
- GCM Tag 保证数据未被篡改或损坏

每个文件的加密数据格式：

```
+-------------------+-----------------------------------+
| Nonce (12 bytes)  | Ciphertext + GCM Tag (16 bytes)   |
+-------------------+-----------------------------------+
```

### 4. 密码验证块

用于在解压前快速检测密码是否正确，无需解压任何实际数据。

**原理**:
1. 压缩时：用验证子密钥加密已知明文 (`"ZBAK_VERIFY_OK!!"`)，将 GCM Tag 和 Nonce 存入文件头
2. 解压时：重新派生验证子密钥，加密同一明文，比较 GCM Tag 是否一致

**安全措施**:
- Tag 比较使用**常量时间比较** (`constant_time_eq`)，防止时序侧信道攻击
- 攻击者无法从 Tag 反推密码 (Argon2id 单向性)
- 已知明文不会降低安全性 (AES-GCM 的 IND-CPA 安全性保证)

**用户体验**: 错误密码在约 200-500ms 内即被检测，无需等待整个文件解压后才发现失败。

### 5. Reed-Solomon 纠错

**算法**: Reed-Solomon 纠删码

| 冗余比例 | 可修复损坏 | 典型场景 |
|----------|------------|----------|
| 5% | ~5% 数据损坏 | 日常备份 (SSD/HDD) |
| 10% | ~10% 数据损坏 | 移动设备 / U盘 |
| 20% | ~20% 数据损坏 | 极端可靠性需求 |

恢复记录附加在文件末尾，不影响正常读取。解压时自动检测是否有恢复记录，
数据损坏时自动尝试修复。

### 6. 文件名加密

开启后，索引区使用专用子密钥 (HKDF "zbak-index") 加密。

**保护内容**:
- 文件名 / 完整路径
- 文件大小、时间戳
- 目录结构

**效果**: 没有密码的情况下，攻击者无法得知压缩包内有哪些文件。


## 三级防护体系

SecureZip 的数据保护分为三个层级，每一层独立工作：

### 第一级: GCM 认证标签

每个文件的 AES-256-GCM Tag 可以检测：
- 单个比特的翻转 (bit-flip)
- 数据截断
- 恶意篡改

**检测粒度**: 逐文件。一个文件损坏不影响其他文件的检测。

### 第二级: Reed-Solomon 纠错

当 GCM Tag 检测到损坏后，Reed-Solomon 可以修复：
- 随机位错误
- 坏扇区 (磁盘老化)
- 部分数据丢失

**修复范围**: 取决于冗余比例设置 (5%/10%/20%)。

### 第三级: Non-Solid 结构

zbak 采用非固实 (non-solid) 压缩架构：
- 每个文件独立压缩 + 独立加密
- 文件 A 损坏不影响文件 B 的解压
- 可以单独提取任意文件 (无需解压整个包)

对比固实压缩 (solid):
- 7z 默认固实模式下，任何位置的损坏都可能导致整个压缩包不可读
- zbak 的损坏影响被限制在受损的单个文件块内


## .zbak 文件头格式

96 字节固定头，任何开发者都可以解析：

```
偏移    大小    字段              说明
[0-3]   4B     MAGIC            "ZBAK" (0x5A42414B)
[4]     1B     VERSION          格式版本 (当前=1)
[5-6]   2B     FLAGS            标志位 (加密/文件名加密/恢复记录/分块)
[7]     1B     LEVEL            压缩级别
[8-23]  16B    SALT             Argon2id Salt (128-bit)
[24-27] 4B     FILE_COUNT       文件数量
[28-95] 68B    扩展区            偏移/大小/验证 Tag/Nonce 等
```

标志位定义:

| Bit | 值 | 含义 |
|-----|------|------|
| 0 | `0x0001` | 内容已加密 |
| 1 | `0x0002` | 文件名已加密 |
| 2 | `0x0004` | 有恢复记录 |
| 3 | `0x0008` | WebDAV 分块模式 |

算法标识 `0x01` = Zstd + AES-256-GCM + Argon2id + Reed-Solomon。
这意味着任何开发者读到文件头后就知道该用哪些标准算法解密，不存在私有算法。


## WebDAV 安全

### 传输安全

- 所有数据在上传前已完成加密 (Zstd 压缩 → AES-256-GCM 加密)
- WebDAV 服务器只存储密文，即使服务器被入侵也无法获取明文
- 建议使用 HTTPS URL 以保护 WebDAV 凭据 (用户名/密码)

### 分块上传安全

- 每个 50MB chunk 是加密后的数据，单独拿到任何 chunk 都无法解密
- manifest.json 只包含 chunk 列表和校验和，不包含加密密钥信息
- 断点续传通过 HEAD 请求检查已上传的 chunk，不需要重新加密


## EXIF 隐私保护

照片备份支持 4 级 EXIF 处理 (exifStripLevel 参数):

| 级别 | 值 | 保留内容 | 移除内容 | 适用场景 |
|------|-----|----------|----------|----------|
| 保留全部 | 0 | 所有 EXIF 数据 | 无 | 个人私密备份 |
| 移除位置 | 1 | 拍摄参数、时间 | GPS 坐标 | 分享前处理 |
| 仅保留基本 | 2 | 图片尺寸、格式 | GPS、拍摄参数、设备信息 | 高隐私需求 |
| 完全移除 | 3 | 无 EXIF | 所有 EXIF 元数据 | 最高隐私 |

EXIF 数据可能包含的敏感信息：
- **GPS 坐标**: 精确的拍摄地点（家庭住址、工作地点）
- **设备信息**: 手机型号、镜头参数
- **拍摄时间**: 精确到秒的时间戳
- **缩略图**: 可能包含裁剪前的原图缩略图


## 密码强度计算

`calculatePasswordStrength` 函数评估密码的抗暴力破解能力。

### 评分规则

| 检查项 | 条件 | 加分 |
|--------|------|------|
| 基本长度 | >= 8 字符 | +1 |
| 较长 | >= 12 字符 | +1 |
| 小写字母 | 包含 a-z | +1 |
| 大写字母 | 包含 A-Z | +1 |
| 数字 | 包含 0-9 | +1 |
| 特殊字符 | 包含非字母数字字符 | +1 |

原始分 (0-6) 映射到展示分 (0-4):

| 原始分 | 展示分 | 描述 |
|--------|--------|------|
| 0-1 | 0 | 非常弱 |
| 2 | 1 | 弱 |
| 3 | 2 | 中等 |
| 4 | 3 | 强 |
| 5-6 | 4 | 非常强 |

### 推荐密码策略

- **最低要求**: 12 位以上，包含大小写字母和数字 (强度 >= 3)
- **推荐**: 16 位以上，包含特殊字符 (强度 = 4)
- **最佳实践**: 使用 `generateRandomPassword(20, true)` 生成的随机密码，配合密码本管理

### 随机密码生成

`generateRandomPassword` 使用密码学安全的随机数生成器 (`rand::thread_rng()`)，
并保证每个字符类别至少出现一次 (Fisher-Yates 洗牌避免固定位置模式)。


## 安全设计决策记录

### 为什么选择 Argon2id 而不是 PBKDF2/bcrypt?

- **PBKDF2**: 计算开销仅依赖 CPU，GPU/ASIC 加速效果显著
- **bcrypt**: 内存开销固定 4KB，不足以抵抗现代 GPU 攻击
- **Argon2id**: 同时抗 GPU (高内存) 和侧信道攻击 (id 混合模式)，OWASP 2024 首选推荐

### 为什么每个文件独立密钥?

如果所有文件共用一个密钥 + 不同 nonce：
- AES-GCM 在同一密钥下安全使用的 nonce 数量有上限 (~2^32)
- 单文件密钥泄露等于全部泄露

HKDF 子密钥方案：
- 每个文件独立密钥 + 独立随机 nonce，nonce 碰撞概率极低
- 单文件密钥泄露不影响其他文件

### 为什么不加密文件大小?

zbak 文件头中存储了文件数量 (4 字节)，但每个文件的原始大小和压缩大小存储在索引区。
当开启文件名加密时，索引区被整体加密，文件大小信息也随之加密。

未开启文件名加密时，索引区明文存储，文件大小可见。
这是一个有意的设计取舍：很多场景下用户只需要保护文件内容而不需要隐藏文件名。

### 为什么密码验证块使用加密而不是 HMAC?

两种方案都安全。选择加密方式 (重新加密已知明文并比较 Tag) 的原因是：
- 无需额外引入 HMAC 库
- 复用已有的 AES-GCM 实现，减少代码量
- GCM Tag 本身就是认证标签，满足验证需求
